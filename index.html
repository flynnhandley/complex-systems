The Paper

In 1998, Richard Cook published a short paper that would become one of the most widely referenced works in systems safety. “How Complex Systems Fail” is eighteen principles — each just a paragraph or two — that describe why complex systems break, why we misunderstand those breakdowns, and why our instinct to assign blame actively prevents us from learning.

Originally written about healthcare, the paper applies to any complex socio-technical system: aviation, nuclear power, financial trading platforms — and software engineering.

Richard Cook (1953–2022)

Richard Cook was a physician, anesthesiologist, and safety researcher whose work changed how hospitals think about error and patient safety. Rather than blaming individuals, he focused on how complex healthcare systems fail under pressure — and how frontline clinicians constantly adapt to keep patients safe. His research in hospitals and critical care helped shape modern thinking on resilience, human factors, and safer healthcare operations.

Section 1

The Comms Service

The Comms API is a customer communications platform. Its job is straightforward: send emails and SMS messages to customers. SMS delivery routes through an external provider called Whisper.

At its core, a communications service needs to receive a request, persist it, process it, and send it. A well-designed version of this service could be simple, durable, and easy to operate.

Section 2

What Simple Looks Like

A communications service does not need to be complex. One API, one storage account, one service bus, and a single integration point with the SMS provider. Simple, durable, easy to operate.

A simpler design — one service, durable storage, easy to operate

Simple Design Principles

  • One Comms API — single service to understand and operate
  • Service Bus for async messaging — decouples ingest from delivery
  • Durable, self-cleaning storage — messages expire, nothing persists indefinitely
  • Single integration point with Whisper — one place where SMS leaves the system
Section 3

What It Actually Looks Like

The Comms API comprises 27 Azure Function Apps. They communicate through a mix of direct synchronous service calls, Service Bus queues and topics, Step Function orchestration, Table Storage, and Blob Storage. The system spans ingest, validation, enrichment, templating, routing, rate limiting, throttling, personalisation, orchestration, delivery, and status tracking.

It was designed for scale and flexibility. It was difficult to understand, test, and safely operate.

Section 4

The Incidents

The team that originally built the Comms API was disbanded. Ownership was handed to a new team that had no involvement in its development, no operational runbooks, and no training on the system's failure modes. Within a month, three customer-impacting incidents occurred.

Incident 1 — Production

Failed Batch Job

Hundreds of thousands of SMS messages were queued for processing. The batch job loaded every message into memory, processed them, attempted to send them all, and only then tried to mark them as complete. The process crashed partway through, and messages that had already been sent were never marked as completed.

Multiple compounding issues made this inevitable:

1
32-Bit Runtime In Production
Production ran a 32-bit runtime. Non-production ran 64-bit. Even after scaling the worker to 16 GB, the application could only access 4 GB of memory.
2
Different Code In Production
Dependency injection swapped the core sending service in non-production for a mock. The real LINQ batch logic — the code that actually ran in production — was never tested.
3
Not Streaming — Full Copy To Memory
The code used .GetAllAsync(), .Where(), .ToList(), and .Select().ToList() — creating multiple copies of the entire dataset in memory before processing.
4
No Per-Message Checkpointing
Messages were only deleted after the entire batch was sent. A crash at any point meant nothing was marked as complete — even messages that had already been delivered.

Production Code — The Dangerous Batch Pattern

Production — Unsafe Batch 32-bit · Untested 
public async Task Run()
{
    // Load ALL pending messages into memory
    var all = await _tableStorage
        .GetAllAsync("SmsPending");

    // Filter + copy into new list
    var messages = all
        .Where(r => r.Channel == "SMS")
        .ToList();                    // copy 1

    // Build SMS objects — another full copy
    var smsList = messages
        .Select(r => new Sms
        {
            Id    = r.RowKey,
            Mobile = r.Mobile,
            Body  = r.Body
        })
        .ToList();                    // copy 2

    // Send everything
    await _whisperClient
        .SendBatchAsync(smsList);

    // Only NOW delete — if crash above,
    // nothing is marked as sent
    await _tableStorage
        .DeleteBatchAsync(messages);
}
Non-Production — Mock 64-bit · "Tested" 
public class MockSmsSendingService
    : ISmsSendingService
{
    public Task Run()
    {
        // No table storage queries
        // No LINQ chains
        // No memory allocation
        // No Whisper calls
        // No batch deletes

        return Task.CompletedTask;

        // The test "passes" but exercises
        // none of the production logic.
    }
}

The Dependency Injection Swap

The same codebase registered a completely different service depending on the environment. In production, the real service ran all the LINQ batch logic. In non-production, the mock bypassed it entirely.

✓ Non-Production "Passed" 64-bit runtime, no memory limit hit. Mock service returns immediately. Core business logic never executes. Performance testing gave false confidence.

"Successful testing can create false confidence when environments differ."

How Complex Systems Fail — Principle 7
Incident 2 — Non-Production

Test Messages Sent To Real Customers

During the migration to a new platform, a tester was running performance tests. They queued approximately 100,000 messages and then manually disabled the Azure Function that sends messages — they were testing the queuing service, not the sending service.

Later that evening, an offshore platform engineer needed to do routine work in the same environment. They restarted the function. The moment it started, it picked up the 100,000 queued messages and sent them to real customers — because the test environment used a copy of production data.

Latent Failures Exposed By One Click

  • Production customer data in the test environment
  • No synthetic or anonymised test data
  • No safe startup mode — the function immediately processed the full backlog
  • No feature flag or environment guard to prevent real sends
  • No queue TTL — 100,000 messages sat waiting indefinitely
  • No documentation warning that restarting the function would process stale queues

"The person clicked the button. The system made that dangerous."

Key takeaway — one click exposed many latent failures already present
Incident 3 — Production

Stale Messages Resurface During Soft Launch

The platform team was migrating the Comms API to new infrastructure intended to solve some of the issues identified during the previous incidents. They performed a soft launch — deploying the application and infrastructure to production without routing any new traffic to it yet.

However, the new instance of the microservice connected to the same storage. It picked up the unacknowledged messages left over from Incident 1 — messages that had been sent but never marked as complete when the process crashed. Two months had passed. Customers received bill reminders that were no longer valid.

The Chain Of Causation

  • Incident 1: batch job crashes → sent messages not marked as complete
  • No TTL on messages → stale records persist for months
  • New instance connects to same storage → picks up orphaned messages
  • No startup safety check → immediately processes everything it finds
  • Customers receive outdated, invalid reminders

"Change introduces new forms of failure. Every change to the system creates new pathways for accidents."

How Complex Systems Fail — Principle 9
Section 5

Key Learnings — How Complex Systems Fail

Richard Cook's paper "How Complex Systems Fail" describes eighteen principles about failure in complex systems. These incidents illustrate many of them. Here are the most relevant.

Catastrophe Requires Multiple Failures

No single factor caused any of these incidents. Each resulted from a combination of conditions — each individually insufficient to cause harm, but together creating the environment for cascading failure.

Contributing Factors Across All Three Incidents
Principle 3
Catastrophe Requires Multiple Failures
Overt catastrophic failure occurs when small, apparently innocuous failures join to create opportunity for a systemic accident. Each of these small failures is necessary to cause catastrophe, but only the combination is sufficient.

Hindsight Bias

After the incidents, people said "It was obvious," "They should have known," and "They should have asked." These statements are products of hindsight bias — knowing the outcome distorts our judgement of decisions made before the outcome was known.

After The Incident
  • "It was obvious."
  • "They should have known."
  • "They should have asked."
  • "Anyone would have checked."
vs
Before The Incident
  • Routine environment restart
  • No warning or guard rails
  • No documentation of risk
  • Looked like every other restart
Principle 16
Hindsight Biases Post-Accident Assessments Of Human Performance
Knowledge of the outcome makes it seem that participants "should have known" that the decisions they were making would lead to failure. Hindsight bias remains the primary obstacle to accident investigation.

Attribution To A "Root Cause" Is Wrong

The instinct after these incidents was to blame the offshore engineer who restarted the Azure Function. But that individual did not create any of the conditions that made the restart dangerous. They interacted with a system already primed for failure.

The Engineer Did Not Create

  • Production data in the test environment
  • Stale unacknowledged queues
  • Unsafe startup behaviour (immediate processing on boot)
  • Missing environment controls or feature flags
  • The weak ownership transition between teams
  • The 27-microservice architecture
Principle 17
New Understanding Of "Human Error" Is Needed
The common approach to the failure of complex systems treats human performance as the cause of failure. But human operators are actually the adaptable element of the system, providing flexibility and creating safety. Failure is always a property of the system, not of the individuals within it.

Complex Systems Run In Degraded Mode

Before any incident occurred, the Comms API was already running with multiple latent defects: a runtime mismatch, untested production code paths, production data in test, no message TTL, and no safe startup behaviour. The system appeared to be working, but it was operating in a degraded state the entire time.

Principle 5
Complex Systems Run In Degraded Mode
Complex systems are always running as broken systems. They keep functioning because they contain redundancies and because people can make them function despite the presence of many flaws. System operations are dynamic, with components failing and being replaced continuously.

Safety Is Created By Systems, Not Slogans

Telling people to "be more careful" does little. Real safety comes from engineering controls that make dangerous actions difficult or impossible.

  • Prod / non-prod parity — Same runtime, same code paths, same architecture. No mock services that bypass core logic.
  • Safe synthetic test data — Never use copies of production customer data in test environments.
  • Idempotent sends — Sending the same message twice must produce the same result.
  • Queue TTL and cleanup — Messages expire after a defined period. Stale data does not persist indefinitely.
  • Safe startup modes — Services do not immediately process backlogs on restart. Require explicit activation.
  • Stream, don't batch — Process one message at a time. Checkpoint after each. Never load everything into memory.
  • Better observability — Alerts on queue depth, memory usage, message age. Make the system's state visible.
  • Clear ownership and training — Teams that operate a service must understand its failure modes.

"Stream, send, checkpoint. Don't batch, blast, and pray."
